# 音乐运动节拍器App需求文档

## 1. 项目概述

### 1.1 产品定位
一款基于音乐BPM的运动节拍器应用，通过音乐节拍引导用户进行跑步、散步等运动，帮助用户保持稳定的运动节奏。

### 1.2 核心价值
- 用音乐替代传统滴答声节拍器，提升运动体验
- 根据BPM智能匹配运动形式和速度
- 通过**分享运动节拍/运动状态**，让用户之间可以互相传递、复现彼此的运动节奏
- 支持用户基于参照音乐和BPM，快速找到/配置适合自己的运动节奏
- 歌单功能作为辅助工具，帮助用户组织和管理适合运动的音乐
- 无需音乐播放权限，通过跳转第三方音乐app播放

### 1.3 目标用户
- 跑步爱好者
- 散步健身人群
- 需要稳定运动节奏的用户

## 2. 核心功能需求

### 2.1 功能一：音乐运动形式识别
**问题**：选中的音乐是什么运动形式，跑步还是散步，速度快还是慢？

**功能描述**：
- 用户输入/选择一首歌曲
- App根据歌曲BPM值，自动识别并显示：
  - 运动形式：跑步 / 散步
  - 速度等级：快 / 中 / 慢
  - BPM数值显示

**输出示例**：
- "这首歌适合**慢跑**，速度**中等**（BPM: 140）"
- "这首歌适合**快走**，速度**较慢**（BPM: 110）"

### 2.2 功能二：运动形式歌曲推荐
**问题**：选定的运动形式，有什么歌可以做节拍器？

**功能描述**：
- 用户选择运动形式（跑步/散步）和速度偏好（快/中/慢）
- App从BPM数据库中筛选匹配的歌曲
- 支持多条件筛选：
  - 语言：中文、英文、日文等
  - 歌手/乐队：如"五月天"、"周杰伦"等
  - 音乐风格：流行、摇滚、电子等
  - BPM范围：自定义区间
- 显示推荐歌曲列表，每首显示BPM值和速度提示

### 2.3 功能三：运动节拍/运动状态分享（核心功能）
**问题**：如何让“音乐节拍器的运动感觉”在用户之间传递，而不仅仅是分享一份歌单？

**核心理念**：
- 分享的不是具体哪首歌，而是**一套运动状态/节奏配置**，其他用户可以套在自己喜欢的歌上使用。
- 让“我现在这组跑步节奏很好用”可以通过链接/二维码/模板的方式，传递给朋友或社区。

**功能描述**：
- 定义“运动节拍配置/运动状态模板”，包括：
  - 运动形式（跑步/散步）
  - 目标步频/目标BPM区间
  - 速度感受（慢/中/快）
  - 参考歌曲示例（可选）
- 用户可以：
  - 基于参照音乐和个人反馈，生成自己的“运动状态模板”
  - 一键生成**分享链接/二维码**，发送给朋友或分享到社交平台
  - 浏览他人分享的运动状态模板，一键“套用”，用自己的歌去匹配这个节奏
- 当接收方打开分享时：
  - 看到这套运动状态的说明（适合什么运动、什么速度感受、推荐BPM范围等）
  - App基于本地或云端BPM库，推荐与该状态匹配的歌曲
  - 接收方可以用自己的歌单/喜欢的歌，按这个BPM节奏去运动

**与歌单的关系**：
- 歌单只是承载节奏的一个“容器”，不再是核心目标。
- 核心是：**先定义和分享节奏/运动状态，再用歌单去实现这个节奏**。

**用户分享场景示例**：
1. 用户A调了一套“5’30配速慢跑、轻松聊天节奏”的状态模板，配了几首歌做示例。
2. 用户A分享给用户B（链接/二维码）。
3. 用户B打开后：
   - 先体验这套节奏（用参照音乐或自己的歌）。
   - 再用App的搜索功能，按这套BPM范围找自己喜欢的歌。
4. 用户B也可以在此基础上稍微调整后，再分享出去，形成“节拍器玩法”的传播。

## 3. 用户使用流程

### 3.1 初次使用引导流程

1. **参照音乐体验**
   - App提供预设的参照音乐文件（本地音频）
   - 引导用户完成一次基础运动体验
   - 让用户感受音乐节拍与运动节奏的匹配

2. **用户反馈收集**
   - 用户反馈对当前速度的感受：
     - 太快了
     - 太慢了
     - 刚好

3. **智能推荐**
   - App根据用户反馈和参照音乐的BPM值
   - 从数据库中推荐BPM相近或调整后的歌曲
   - 显示速度对比提示：
     - "与参照音乐速度**差不多**"
     - "比参照音乐**更快**（+15 BPM）"
     - "比参照音乐**更慢**（-10 BPM）"

4. **歌单制作（辅助）**
   - 用户从推荐列表中选择喜欢的歌曲
   - 添加到个人运动歌单，用来承载某个运动状态/节拍配置
   - 支持创建多个歌单（如"晨跑歌单"、"夜跑歌单"），但歌单本身不是核心目标

### 3.2 日常使用流程

1. 选择运动形式 → 2. 浏览/搜索歌曲 → 3. 查看速度提示 → 4. 添加到歌单 → 5. **导入歌单到音乐app播放**

### 3.3 歌单导入播放流程（核心流程）

1. **创建/选择歌单**
   - 用户创建运动歌单或选择已有歌单

2. **选择导入方式**
   - 点击"在音乐App中播放"按钮
   - 选择目标音乐平台（网易云/QQ音乐/华为/酷狗/Apple Music）

3. **自动导入歌单**
   - App尝试通过API或Deep Link导入歌单
   - 如果成功，直接跳转到音乐app并开始播放
   - 如果失败，提供降级方案（导出文件/分享链接/复制文本）

4. **连续播放体验**
   - 用户在音乐app中连续播放整个歌单
   - 无需每首播放完返回App点击下一首
   - 实现真正的"一键导入，连续播放"体验

## 4. 详细功能说明

### 4.1 音乐搜索功能

**搜索方式**：
- 关键词搜索：歌曲名、歌手名
- 筛选搜索：
  - 运动形式（跑步/散步）
  - 速度等级（快/中/慢）
  - 语言类型
  - 歌手/乐队
  - 音乐风格
  - BPM范围

**搜索结果展示**：
- 歌曲信息：歌名、歌手、专辑封面
- BPM数值
- 运动形式标识
- 速度对比提示（相对于参照音乐或用户上次使用的音乐）

### 4.2 歌单管理功能

**创建歌单**：
- 支持创建多个歌单
- 自定义歌单名称、描述
- 设置歌单对应的运动形式

**编辑歌单**：
- 添加/删除歌曲
- 调整歌曲顺序
- 编辑歌单信息

**分享歌单**：
- 生成歌单分享链接/二维码
- 支持导出歌单信息（文本格式）
- 其他用户可导入分享的歌单

### 4.3 歌单导入音乐app播放（核心功能）

**功能目标**：
- **解决单曲跳转体验差的问题**：用户不可能每播放完一首歌就返回App点击下一首
- **实现整个歌单的连续播放**：一次性导入歌单到音乐app，用户可在音乐app中连续播放

**实现方式（按优先级）**：

#### 方式一：API创建播放列表（最理想）
- **Spotify**：通过Spotify Web API创建播放列表
  - 需要用户OAuth授权
  - 调用API创建播放列表并添加歌曲
  - 跳转到Spotify App播放
- **Apple Music**：通过MusicKit API创建播放列表（iOS）
  - 需要用户授权
  - 使用MusicKit框架创建播放列表
- **其他平台**：调研各平台开放API支持情况

#### 方式二：Deep Link跳转（次优）
- **网易云音乐**：通过URL Scheme传递歌单信息
  - 格式：`orpheus://playlist?name=xxx&songs=xxx`
  - 需要调研网易云是否支持歌单参数
- **QQ音乐**：通过URL Scheme传递歌单信息
  - 格式：`qqmusic://playlist?xxx`
  - 需要调研QQ音乐Deep Link格式
- **华为音乐**：通过华为运动健康App选择歌单
  - 跳转到华为运动健康App
  - 选择歌单后自动跳转到华为音乐播放

#### 方式三：导出标准格式文件（通用方案）
- **M3U格式**：导出为.m3u文件
  - 大部分音乐播放器支持
  - 用户手动导入到音乐app
- **平台专用格式**：导出为各平台可识别的格式
  - 网易云音乐格式
  - QQ音乐格式
  - 其他平台格式

#### 方式四：分享链接/二维码（社交分享）
- **生成分享链接**：将歌单信息编码到URL
  - 格式：`https://your-app.com/playlist/{encoded_data}`
  - 用户扫码或点击链接
  - 在App中打开后跳转到音乐平台
- **生成二维码**：方便用户扫码导入

#### 方式五：复制文本（降级方案）
- **复制歌单信息**：将歌单歌曲列表复制到剪贴板
  - 格式：`歌曲名 - 歌手名`（每行一首）
  - 用户粘贴到音乐app搜索
  - 手动添加到播放列表

**多平台支持策略**：

| 平台        | 优先级 | 实现方式        | 备注                 |
|-------------|--------|-----------------|----------------------|
| 网易云音乐  | 高     | Deep Link/API   | 需调研开放平台      |
| QQ音乐      | 高     | Deep Link/API   | 需调研开放平台      |
| 华为音乐    | 中     | 通过运动健康App | 华为用户            |
| 酷狗音乐    | 中     | Deep Link       | 需调研              |
| Apple Music | 高     | API方式         | 仅iOS，需要授权     |

**降级策略**：
```
尝试API方式
  ↓ 失败
尝试Deep Link方式
  ↓ 失败
导出M3U文件
  ↓ 失败
生成分享链接
  ↓ 失败
复制歌单文本到剪贴板
```

**用户体验要求**：
- 导入成功率 > 80%
- 导入操作步骤 ≤ 3步
- 支持离线导出（M3U文件）
- 提供清晰的操作指引和错误提示

## 6. 界面需求

## 7. 技术实现要点

### 7.1 核心技术栈
- 移动端框架：React Native / Uniapp / Flutter / 原生开发
- 数据库：SQLite（本地） + 云端数据库（同步）
- BPM数据：需要建立和维护BPM数据库

### 7.2 关键功能实现

1. **BPM数据库**
   - 需要收集大量歌曲的BPM数据
   - 支持手动录入和批量导入
   - 定期更新和维护

2. **歌单导入音乐app（核心功能）**
   - **API方式**：
     - 集成Spotify Web API（OAuth授权流程）
     - 集成Apple Music API（MusicKit框架）
     - 调研其他平台开放API
   - **Deep Link方式**：
     - 网易云音乐URL Scheme
     - QQ音乐URL Scheme
     - 其他平台Deep Link协议
   - **文件导出**：
     - M3U格式生成
     - 平台专用格式转换
   - **分享功能**：
     - 歌单信息编码/解码
     - 二维码生成
     - 分享链接处理
   - **降级策略**：
     - 多方案自动降级
     - 错误处理和用户提示

3. **参照音乐**
   - 使用本地音频文件（无版权问题）
   - 或使用免版权的示例音乐
   - 提供多个参照音乐供选择

4. **速度对比算法**
   - 计算目标歌曲BPM与参照音乐BPM的差值
   - 设定阈值判断"差不多"（±5 BPM）、"更快"、"更慢"

- Android 8.0+
- 支持主流音乐平台跳转

## 11. 风险评估

1. **BPM数据获取**：需要大量人力收集和维护BPM数据
2. **歌单导入成功率**：不同平台API/Deep Link支持可能不稳定
   - 需要调研各平台API文档和Deep Link格式
   - 需要处理各平台的授权流程（OAuth等）
   - 需要设计完善的降级策略
3. **版权问题**：确保不涉及音乐播放版权
4. **用户体验**：
   - 歌单导入操作要简单直观
   - 需要清晰的错误提示和操作指引
   - 支持离线导出功能

4. **【核心指标】歌单导入音乐app成功率 > 80%**
